## SECTION 1 — HDS — HÉBERGEUR DE DONNÉES DE SANTÉ

### 1.1 Ce que c'est

| Aspect | Détail |
|--------|--------|
| **Définition** | La certification HDS (Hébergeur de Données de Santé) est un agrément délivré aux hébergeurs de données de santé qui respectent des exigences strictes de sécurité, de confidentialité et de disponibilité des données. |
| **Base légale** | Article L.1111-8 du Code de la Santé Publique |
| **Décret d'application** | Décret n°2018-137 du 26 février 2018 relatif à l'hébergement des données de santé |
| **Organisme certificateur** | Organisme accrédité COFRAC (audit annuel) |
| **Sanction** | Non-conformité = interdiction d'exercice de l'hébergement + amendes jusqu'à 150 000€ |

### 1.2 Ce que la certification HDS exige

| Exigence | Détail |
|----------|--------|
| **Sécurité physique** | Locaux sécurisés, accès contrôlé, environnement protégé (incendie, inondation, intrusion) |
| **Sécurité logique** | Chiffrement des données (au repos et en transit), authentification forte, journalisation des accès |
| **Continuité d'activité** | Plan de reprise d'activité (PRA), sauvegardes régulières, redondance des infrastructures |
| **Personnel** | Formation des équipes, confidentialité, habilitation, contrôle d'accès |
| **Audit** | Audit annuel obligatoire par un organisme accrédité COFRAC |
| **Traçabilité** | Traçabilité exhaustive de toutes les actions sur les données |

### 1.3 Qui est concerné par HDS

| Acteur | Obligation HDS |
|--------|----------------|
| **Hébergeurs** | ✅ Obligatoire (OVHcloud, AWS, Microsoft Azure, Google Cloud) |
| **Éditeurs de logiciels médicaux** | ✅ Obligatoire pour héberger leurs données |
| **Cabinets médicaux** | ✅ Obligatoire pour stocker les données patients |
| **Développeurs d'applications médicales** | ✅ Obligatoire s'ils traitent des données de santé |

### 1.4 Conséquences d'une non-conformité HDS

| Conséquence | Détail |
|-------------|--------|
| **Amendes** | Jusqu'à 150 000€ pour l'hébergeur |
| **Interdiction** | Interdiction d'exercer l'hébergement de données de santé |
| **Responsabilité** | Engagement de la responsabilité civile du praticien utilisateur |
| **Risque patient** | Exposition des données patients à des violations de sécurité |

### 1.5 La spécificité française HDS vs HIPAA (US)

| Critère | HDS (France) | HIPAA (US) |
|---------|--------------|------------|
| **Base légale** | Code de la Santé Publique | Health Insurance Portability and Accountability Act |
| **Juridiction** | France (UE) | États-Unis |
| **Cloud Act** | ✅ Non applicable (hébergement France) | ❌ Applicable (données saisissables par le gouvernement US) |
| **Audit** | Annuel (COFRAC) | Périodique (HHS) |
| **Amendes** | Jusqu'à 150 000€ | Jusqu'à 1,5M$ |
| **Sanction pénale** | Oui (interdiction d'exercice) | Oui (amendes pénales) |
| **Responsabilité** | Responsabilité de l'hébergeur | Responsabilité du "covered entity" |

### 1.6 Formulation standard pour les contenus SAYDO

> *"SAYDO est hébergé sur OVHcloud, certifié Hébergeur de Données de Santé (HDS) selon les exigences de l'article L.1111-8 du Code de la Santé Publique. Aucun traitement ne sort du territoire français. Audit annuel COFRAC, documents disponibles sur demande."*


## SECTION 2 — RGPD — RÈGLEMENT GÉNÉRAL SUR LA PROTECTION DES DONNÉES

### 2.1 Ce que c'est

| Aspect | Détail |
|--------|--------|
| **Définition** | Le RGPD (Règlement Général sur la Protection des Données) est le règlement européen qui encadre le traitement des données personnelles. |
| **Base légale** | Règlement (UE) 2016/679 du Parlement européen et du Conseil du 27 avril 2016 |
| **Entrée en vigueur** | 25 mai 2018 |
| **Applicabilité** | Toute organisation qui traite des données personnelles de citoyens européens |
| **Sanction** | Amendes jusqu'à 20 millions d'€ ou 4% du chiffre d'affaires mondial annuel (le plus élevé) |

### 2.2 Principes clés du RGPD

| Principe | Détail | Application SAYDO |
|----------|--------|-------------------|
| **Licéité** | Traitement basé sur une base légale (consentement, contrat, obligation légale) | ✅ Base de données patients sécurisée |
| **Loyauté** | Information transparente du patient | ✅ Patient informé |
| **Transparence** | Information claire sur les traitements | ✅ Politique de confidentialité |
| **Minimisation** | Collecte des données strictement nécessaires | ✅ Seulement les données cliniques utiles |
| **Exactitude** | Données exactes et mises à jour | ✅ Saisie automatique = erreur réduite |
| **Limitation de conservation** | Données conservées le temps nécessaire | ✅ Conformité HDS |
| **Intégrité et confidentialité** | Sécurité des données | ✅ Chiffrement + HDS |
| **Responsabilité (Accountability)** | Documentation et preuve de conformité | ✅ Audits HDS annuels |

### 2.3 Droits des patients (personnes concernées)

| Droit | Détail | Application SAYDO |
|-------|--------|-------------------|
| **Droit d'accès** | Le patient peut demander ses données | ✅ Possibilité d'export |
| **Droit de rectification** | Le patient peut corriger ses données | ✅ Accès aux données |
| **Droit à l'effacement** | Le patient peut demander la suppression de ses données | ✅ Conformité RGPD |
| **Droit à la limitation** | Le patient peut limiter le traitement | ✅ Conformité RGPD |
| **Droit à la portabilité** | Le patient peut récupérer ses données | ✅ Export possible |
| **Droit d'opposition** | Le patient peut s'opposer au traitement | ✅ Conformité RGPD |

### 2.4 Obligations du praticien utilisateur

| Obligation | Détail |
|------------|--------|
| **Information** | Le patient doit être informé du traitement de ses données |
| **Consentement** | Le consentement du patient doit être recueilli |
| **Sécurité** | Les données doivent être protégées (chiffrement, accès contrôlé) |
| **Conformité du sous-traitant** | Le praticien doit s'assurer que SAYDO (sous-traitant) est conforme |
| **Registre** | Le praticien doit tenir un registre des traitements |

### 2.5 Formulation standard pour les contenus SAYDO

> *"SAYDO est conforme au RGPD. Le praticien reste responsable de la gestion des données patients, et SAYDO agit comme sous-traitant conforme. L'hébergement HDS et le chiffrement des données garantissent la protection des informations personnelles de santé."*


## SECTION 3 — CLOUD ACT — LA MENACE AMÉRICAINE

### 3.1 Ce que c'est

| Aspect | Détail |
|--------|--------|
| **Définition** | Le Cloud Act (Clarifying Lawful Overseas Use of Data Act) est une loi américaine de 2018 qui autorise le gouvernement américain à saisir des données détenues par des entreprises américaines, OÙ QU'ELLES SE TROUVENT dans le monde. |
| **Base légale** | Public Law 115-141, Section 10501 |
| **Entrée en vigueur** | 2018 |
| **Applicabilité** | Toute entreprise américaine ou entreprise étrangère avec une présence substantielle aux États-Unis |

### 3.2 Pourquoi c'est une menace pour les praticiens français

| Menace | Détail |
|--------|--------|
| **Saisie des données** | Le gouvernement américain peut exiger l'accès aux données des patients français |
| **Absence de recours** | Le praticien français n'a pas de recours juridique contre une saisie américaine |
| **Violation du RGPD** | Une saisie Cloud Act constitue une violation du RGPD (transfert de données hors UE sans garanties) |
| **Risque de non-conformité HDS** | Les données quittent le territoire français → violation HDS |
| **Perte de confiance** | Les patients ne sont pas informés que leurs données peuvent être saisies par un gouvernement étranger |

### 3.3 Quelles entreprises sont concernées par le Cloud Act

| Entreprise | Concernée par le Cloud Act ? | Raison |
|------------|------------------------------|--------|
| Microsoft (Nuance) | ✅ Oui | Siège aux États-Unis |
| Google (Nabla ?) | ✅ Oui | Siège aux États-Unis |
| Amazon (Amazon HealthScribe) | ✅ Oui | Siège aux États-Unis |
| OVHcloud (SAYDO) | ❌ Non | Siège en France, pas de présence substantielle aux US |

### 3.4 Conséquences pour SAYDO

| Conséquence | Détail |
|-------------|--------|
| **Avantage concurrentiel** | SAYDO est la seule solution française HDS → non soumise au Cloud Act |
| **Argument de vente** | Les données des patients ne quittent jamais la France |
| **Conformité** | SAYDO est conforme HDS, RGPD, et Code de la Santé Publique |
| **Sécurité juridique** | Le praticien ne risque pas de violation du RGPD |

### 3.5 Formulation standard pour les contenus SAYDO

> *"Nuance, Nabla, Abridge et Amazon HealthScribe sont des solutions américaines soumises au Cloud Act. Un praticien français qui les utilise expose potentiellement ses données patients à des juridictions étrangères — une violation possible du RGPD et du cadre HDS. SAYDO est le seul concurrent direct à ne pas avoir ce problème structurel."*
